# 프롬프트 엔지니어링 실습 정리 — 곽민성 포트폴리오 페이지

## 1. 최종 채택한 핵심 프롬프트

### ① Next.js 소개 페이지 생성
> "간단한 ai agent 소개페이지를 next.js로 구성해줘"

### ② 포트폴리오 브랜딩 코치 역할극
> "너는 포트폴리오 브랜딩 코치야. 나에게 한 번에 하나씩 질문해서 내 직무 지향·강점·되고 싶은 인상을 끌어내줘. 표면적 답이 나오면 '왜?'를 더 물어줘. 5~6번 대화 후, 내 [태그라인 1문장 / 키워드 5개 / 추천 컬러 3색]을 정리해줘."

### ③ 실제 이력서 기반 포트폴리오 자료 재구성
> (이력서 PDF 업로드 후) "자 내 사람인에 있는 자기소개서야 이것도 내 포트폴리오에 넣을 자료로 써줘"

### ④ 디자인 스타일 선택지 제시
> "위에서 정한 내 페르소나에 어울리는 '디자인 스타일 1개'와 그 이유, 참고할 만한 레퍼런스 사이트 유형 2개, 과하지 않게 쓸 디자인 기법 2~3개를 추천해줘. 내가 고를 수 있게 선택지로 제시해줘."

### ⑤ 자가 점검용 체크리스트 생성
> "이 프롬프트로 만든 결과물을 내가 스스로 점검할 수 있도록, '3초 룰·일관성·반응형·접근성·보안(API 키 노출)' 관점의 체크 질문 7개를 만들어줘."

## 2. 프롬프트별 설계 근거

| # | 프롬프트 | 왜 이렇게 설계했나 |
|---|---|---|
| ① | Next.js 소개 페이지 요청 | 설명이 아닌 실행 가능한 결과물(코드)을 바로 받기 위해, 프레임워크와 목적을 명시해 산출물 형태를 고정함 |
| ② | 브랜딩 코치 역할극 + '왜?' 반복 질문 | 표면적 자기소개("빛나고 싶다")에서 멈추지 않고, 반복 질문으로 실제 동기와 차별점을 끌어내는 구조를 의도적으로 설계함 |
| ③ | 이력서 업로드 + 자료 재사용 요청 | 브랜딩 세션에서 나온 추상적 키워드를, 실제 이력 데이터와 결합해 과장 없는 근거 기반 카피로 바꾸기 위함 |
| ④ | 디자인 스타일 선택지 요청 | 하나의 정답을 강요받지 않고, 근거가 붙은 여러 옵션 중 직접 비교·선택하도록 의사결정 주도권을 유지하기 위함 |
| ⑤ | 자가 점검 체크리스트 요청 | 결과물을 감으로 판단하지 않고, 5개의 객관적 기준(3초룰·일관성·반응형·접근성·보안)으로 스스로 검증할 수 있게 하기 위함 |

## 3. 검증 결과 요약

- **타입 안정성**: `npx tsc --noEmit` 실행 결과, 코드 전반(page.tsx, layout.tsx, MobileNav.tsx)에서 타입 오류 없음
- **빌드**: 작업 환경 네트워크 제약으로 Google Fonts(`fonts.googleapis.com`) 접근이 막혀 `next build`는 완주하지 못함 — 이는 실제 로컬/배포 환경(정상 인터넷 연결)에서는 발생하지 않는 문제로 확인됨
- **디자인 방향 검증**: 개발자 콘솔 / 미니멀 스위스 그리드 / 시스템 다이어그램 3가지 스타일을 목업으로 미리 비교한 뒤 '시스템 다이어그램' 스타일로 확정, 이후 다크 톤이 과하다는 피드백을 반영해 라이트 톤으로 재조정
- **접근성/반응형**: 자가 점검 체크리스트 적용 결과, 모바일에서 내비게이션 메뉴에 도달할 수 없는 문제(6번 항목)를 발견 → 별도 클라이언트 컴포넌트(`MobileNav.tsx`)로 햄버거 메뉴를 추가해 해결
- **보안**: 코드 내 API 키·시크릿·개인 연락처 하드코딩 없음(이력서의 이메일·전화번호는 마스킹된 상태로만 참고, 실제 값 미삽입) — 단, 최종 제출 전 `grep -rn "API_KEY\|SECRET\|TOKEN" .` 로 직접 재확인 권장
